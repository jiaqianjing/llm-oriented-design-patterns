# LLM-Oriented Design (LOD) Patterns

### 抛弃人类可读性。重铸齿轮法则、反馈回路与工具体系。

---

> **你代码的第一个读者不再是人类，而是 LLM。**
>
> OOP、SOLID 和深层继承是为人类认知而设计的。在 AI 优先的开发时代，它们是*上下文毒药*——膨胀的模块溢出注意力窗口，隐形的间接调用遮蔽调用图谱，耦合的关注点迫使模型加载数千个无关 token。
>
> 本项目是后人类可读性时代的设计宣言，建立在**三大原则**之上。

---

## 速览

将 [`LOD-for-LLM.md`](LOD-for-LLM_CN.md) 复制到你项目的 `CLAUDE.md`、`.cursorrules` 或系统提示中，三大原则即刻生效。

**用于代码审查**: 使用[总结清单](LOD-for-LLM_CN.md#summary-checklist)，依据三大原则审计任何代码库。


## 问题所在

即使拥有百万 token 的上下文窗口，**"中间丢失"**效应依然存在。冗长的、对人类友好的代码稀释了模型注意力，导致推理能力下降和幻觉增多。

例如，一个 665 行的上帝类迫使 LLM 仅为修改一个方法就消耗约 **2,700 个 token**。一个 595 行的单体入口文件将 8 种关注点混杂在一个无法导航的文件中。工厂的工厂模式制造出没有任何模型能可靠追踪的间接调用链。

**你的代码库不仅难以被 AI 阅读，它还在主动让 AI 变蠢。**

### 原则一：上下文管理

> *设计必须在 AI 的注意力与算力限制内起舞。上下文压缩是最高优先级。*

每个模块都是一个**微齿轮**——单一职责的单元，适配在模型的最佳智能窗口内（**4k-8k tokens**，约 200-800 LOC）。单体类被打碎。调用规格取代源码阅读。变体存放在独立文件中，通过简单的字典分发。

| 章节 | 核心思想 |
|---|---|
| **极致拆分** | 将上帝类打碎为单一职责的微齿轮 |
| **动态组装与上下文涌现** | 扁平分发（字典、`importlib`）取代工厂/策略模式 |
| **黑箱化与语义压缩** | 30 行调用规格取代 500+ LOC 的源码阅读 |

**成效：** 要理解整条流水线，LLM 只需读取约 **65 个 token** 的调用规格，而非约 **10,500 个 token** 的源码——缩减 **40 倍**。一个 665 LOC 的上帝类变成 250 LOC 的编排器加上专注的微齿轮。

---

### 原则二：反馈回路

> *静态代码拓扑已死。AI 原生系统通过分层反馈像数字生命体一样进化。*

仅记录指标的系统是盲目的。面向 LLM 的系统会**检测**、**诊断**并**适应**。每个流水线阶段返回带有可量化结果的结构化数据。三层反馈构成一个自我感知的有机体：

```
MICRO  (每步)           NaN loss、梯度爆炸、指标崩溃             →  停止或警告
MACRO  (每轮迭代)       停滞、发散趋势、输出坍缩                 →  建议配置变更
SYSTEM (跨运行)         OOM、超时、崩溃                          →  自动调参重试
```

| 章节 | 核心思想 |
|---|---|
| **自顶向下的目标分解** | 入口点分解为可度量、可测试的子目标 |
| **进化式齿轮编排** | 渐进响应：调参 > 替换组件 > 启用特性 > 退役 |
| **多层反馈网络** | 微观 + 宏观 + 系统级回路构建自愈流水线 |

**成效：** 问题在执行**过程中**被捕获，而非等到运行失败之后。系统自行诊断故障并自适应，无需人工介入。

---

### 原则三：工具化

> *在生成模型的概率世界中，确定性的壁垒必须存在。大型软件系统不能在基础设施上掷骰子。*

LLM 决定**做什么**（路由、参数、策略）。人类封装**怎么做**（数学、I/O、验证）。每段确定性逻辑都被提取为纯函数、经过测试、独立运行的工具，并附有显式契约。带硬约束的 Pydantic 验证器剥夺了 LLM 在基础设施上的即兴发挥自由。

| 组件 | 性质 | 所有者 |
|---|---|---|
| "为此工作负载选择变体 B" | 路由、意图 | LLM（概率性的） |
| 变体 B 的公式本身 | 数学 | 人类编写，已封装 |
| 配置验证（`extra='forbid'`） | 模式强制 | Pydantic（确定性的） |
| 文件 I/O、权重合并、张量运算 | 基础设施 | 独立工具 |

| 章节 | 核心思想 |
|---|---|
| **概率与确定性** | 划清界线，封装确定性的一侧 |
| **基础设施即 LLM 工具** | 将文件 I/O、分布式操作、张量运算提取为独立工具 |
| **零幻觉契约** | 带显式约束的 Pydantic 验证器防止无效配置 |

**成效：** 重复的工具方法变成单一可复用工具。权重合并、检查点保存、归一化——编写一次，测试一次，处处复用。

## 量化效果

在 [oxRL](https://github.com/warlockee/oxRL)（一个包含 18 种算法的后训练框架）上验证。改造前后指标对比：

| 任务 | 改造前 | 改造后 | 缩减倍数 |
|---|---|---|---|
| 修改一个算法变体 | ~2,700 tokens | ~140 tokens | **19x** |
| 理解整条流水线 | ~10,500 tokens | ~260 tokens | **40x** |
| 调试状态持久化 | ~2,700 tokens | ~280 tokens | **9.6x** |
| 修改归一化逻辑 | ~2,300 tokens | ~160 tokens | **14x** |
| 修改处理流水线 | ~2,400 tokens | ~480 tokens | **5x** |

| 指标 | 改造前 | 改造后 |
|---|---|---|
| 最大类（GRPO） | 665 LOC，7 种关注点 | 250 LOC，仅编排逻辑 |
| 主入口 | 595 LOC，8 种关注点 | 120 LOC，纯编排 |
| 权重合并 | 在 2 个类中重复实现 | 单一工具，处处复用 |
| 健康监控 | 无 | 三层反馈（微观 + 宏观 + 系统级） |
| 模块规格 | 无（需阅读源码） | 每个模块均有调用规格 |

## 开发者的新角色

掌握这三大原则的工程师不再是 if-else 的搬运工。我们将成为：

| 原则 | 角色 |
|---|---|
| **上下文管理** | 原子齿轮的提供者 &mdash; 编写小型、单一职责的模块，使其适配 LLM 的注意力窗口 |
| **反馈回路** | 目标-反馈系统的设计者 &mdash; 构建"尝试 > 评估 > 诊断 > 修复 > 重试"循环 |
| **工具化** | 确定性边界的架构师 &mdash; 定义哪些是封装的（工具）、哪些是灵活的（路由） |

> 用人类智慧设计**规则**、**反馈回路**和**工具边界**。
> 让 AI 算力去**组装**、**重构**和**进化**。
